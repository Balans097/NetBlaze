                  <h2>
                    <a onClick="return false;" href="https://opennet.ru/64173-nim" class="nolink">
                      <span id="r_title">Выпуск языка программирования Nim 2.2.6
</span>
                    </a>
                  </h2>
                  <span style="font-size: 10pt;">03.11.2025 22:19 (MSK)</span>
                </td>
              </tr>
            </table>
            <table class="ttxt2">
              <tr>
                <td bgcolor="E9EAD6" id="r_memo" style="padding-top: 15px;" class="chtext">
                  <p>                  <a href="https://nim-lang.org/blog/2025/10/31/nim-226.html">Представлен</a>релиз языка системного программирования <a href="https://nim-lang.org/">Nim 2.2.6</a>.  Обновление вышло спустя шесть месяцев после релиза версии 2.2.4 и включает 141 коммит с исправлениями ошибок и улучшениями производительности. Nim –статически типизированный компилируемый язык программирования с синтаксисом, вдохновлённым Python, и возможностями метапрограммирования на уровне Lisp. Язык компилируется в C, C++ и JavaScript, обеспечивая производительность на уровне C при выразительности высокоуровневых языков.  Код проекта <a href="https://github.com/nim-lang/">поставляется</a>под лицензией MIT.

</p>
                  <p>Ключевые особенности Nim включают мощную систему макросов, работающих на AST во время компиляции, развитую систему обобщённого программирования с концептами, <a href="https://ru.wikipedia.org/wiki/%D0%9C%D1%83%D0%BB%D1%8C%D1%82%D0%B8%D0%BC%D0%B5%D1%82%D0%BE%D0%B4">множественную диспетчеризацию</a>(multiple dispatch), детерминированное управление памятью с поддержкой нескольких стратегий (ARC/ORC, refc, маркировка-и-подметание), встроенную поддержку async/await для асинхронного программирования и FFI для простой интеграции с C/C++/JavaScript. Nim позиционируется как системный язык, подходящий для разработки от встраиваемых систем до веб-серверов, с акцентом на эффективность, безопасность памяти и удобство разработки.


</p>
                  <p>Ключевые <a href="https://github.com/nim-lang/Nim/compare/v2.2.4...v2.2.6">изменения</a>:

<ul>                    <li class="l">Оптимизация move-семантики для полей объектов. Компилятор научился распознавать возможность применения move-операций при возврате полей объектов. Ранее конструкции вида &quot;return obj.field&quot; приводили к копированию данных, теперь компилятор корректно применяет перемещение:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                      <font color="#461b7e">proc getField(obj: MyObject): string =
     return obj.field  # Теперь move вместо copy
</font>                    </pre>                    <p>Это особенно важно для тяжёлых типов данных (строки, последовательности, объекты с ресурсами), где устранение лишнего копирования даёт заметный прирост производительности без изменения кода.

</p></li>                    <li class="l">Полная переработка closure-итераторов с обработкой исключений.
Механизм трансформации замыканий-итераторов полностью переписан, что кардинально улучшило стабильность async-кода с обработкой исключений. Исправлены критические проблемы, включая SIGSEGV при использовании try/except не на верхнем уровне:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                      <font color="#461b7e">iterator problematicIterator(): int {.closure.} =
     for i in 0..10:
       try:
         if i == 5:
           raise newException(ValueError, &quot;test&quot;)
         yield i
       except ValueError:
         discard  # Ранее вызывало SIGSEGV
</font>                    </pre>                    <p>Также решена проблема с некорректным пробросом исключений в finally-блоках внутри closure-итераторов.

</p></li>                    <li class="l">Исправления, связанные с управлением памятью
<ul>                      <li class="l">Устранена фундаментальная проблема в сборщике мусора при обработке циклических структур данных, которая могла приводить к выводу ошибки &quot;Illegal storage access&quot;. Проблема существовала с момента создания языка и проявлялась при сложных графах объектов с взаимными ссылками.

</li>                      <li class="l">Исправлен некорректный порядок уничтожения объектов, который мог приводить к обращению к уже освобождённой памяти:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                        <font color="#461b7e">type
     Resource = object
       data: ptr Data
     Container = object
       resource: Resource
       other: OtherResource

   # Теперь деструкторы вызываются в правильном порядке:
   # сначала other, затем resource
</font>                      </pre></li>                      <li class="l">Сборщик ORC ошибочно помечал окружения некоторых замыканий как циклические, что приводило к задержкам освобождения памяти или утечкам. Теперь анализ циклов работает корректно.

</li>                      <li class="l">Исправлена утечка сокетов в asyncnet при ошибках согласования TLS-соединения:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                        <font color="#461b7e">proc handleClient() {.async.} =
     var socket = await server.accept()
     try:
       await socket.setupSSL()  # При ошибке здесь socket теперь корректно закрывается
     except SSLError:
       discard  # Сокет больше не утекает
</font>                      </pre></li>                      <li class="l">Критические исправления в компиляторе.
<ul>                        <li class="l">Устранена регрессия, при которой глобальные переменные, объявленные внутри процедур с static-параметрами, переинициализировались при каждом вызове:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                          <font color="#461b7e">proc test[N: static int]() =
     var global {.global.}: array[N, int]
     global[0] += 1
     echo global[0]

   test[5]()  # Выводило: 1
   test[5]()  # Должно: 2, но выводило: 1 (было переинициализировано)
</font>                        </pre></li>                        <li class="l">Исправлена генерация кода для глобальных переменных в рекурсивных функциях, которая приводила к неопределённому поведению.

</li>                        <li class="l">Решена древняя проблема генерации некорректного C-кода при использовании конструкторов для глобальных переменных внутри конвертеров:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                          <font color="#461b7e">converter toInt(x: MyType): int =
     let global {.global.} = MyType()  # Генерировал невалидный C-код
     result = global.value
</font>                        </pre></li>                        <li class="l">Устранено падение компилятора при генерации исключений типа Defect и использовании doAssert в определённых контекстах.
</li>                        <li class="l">Улучшения в системе типов
<ul>                          <li class="l">Исправлена невозможность возврата lent-значений из case/if выражений:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                            <font color="#461b7e">proc getBest(a, b: string): lent string =
     if a.len &gt; b.len:
       return a  # Ранее: ошибка компиляции
     else:
       return b
</font>                          </pre></li>                          <li class="l">Устранена проблема с некорректным сохранением значений lent-полей в обобщённых типах:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                            <font color="#461b7e">type
     Wrapper[T] = object
       data: lent T

   proc process[T](w: Wrapper[T]) =
     echo w.data  # Значение теперь корректно сохраняется
</font>                          </pre></li>                          <li class="l">Восстановлена проверка инициализации переменной result для типов с requiresInit, которая была сломана в версии 2.2:
type
     MustInit {.requiresInit.} = object
       value: int

   proc test(): MustInit =
     discard  # Теперь корректно выдаёт ошибку о неинициализированном result
</li>                          <li class="l">Исправлено игнорирование некопируемости (&quot;.noCopy&quot;) базового типа:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                            <font color="#461b7e">type
     Base {.noCopy.} = object
     Derived = object of Base

   var a: Derived
   var b = a  # Теперь корректно запрещено
</font>                          </pre></li>                          <li class="l">Оптимизации производительности
<ul>                            <li class="l">Ускорение оператора &quot;@&quot; для тривиальных типов: устранена критическая деградация производительности при создании последовательностей из массивов простых типов:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                              <font color="#461b7e">let arr = [1, 2, 3, 4, 5]
   let s = @arr  # Было крайне медленно, теперь оптимально
</font>                            </pre></li>                            <li class="l">Оптимизация vmgen.sameConstant: значительно ускорена компиляция за счёт оптимизации сравнения констант в виртуальной машине компилятора и сокращения операций выделения памяти.
</li>                            <li class="l">Разыменование результата cast в одиночном выражении больше не вызывает ненужное копирование:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                              <font color="#461b7e">let data = cast[ptr MyType](address)[]  # Теперь без копирования
</font>                            </pre></li>                            <li class="l">Исправления в бэкенде для компиляции в JavaScript:
<ul>                              <li class="l">=destroy для не-var типов: исправлена генерация деструкторов, которая ранее приводила к ошибкам компиляции. 
</li>                              <li class="l">cast[char] для значений &gt;255: теперь корректно выполняется усечение, как в C-backend. 
</li>                              <li class="l">Концепты в varargs: устранён вывод ошибки &quot;internal error&quot; при передаче концептов в varargs. 

</li>                              <li class="l">В бэкенде для компиляции в C++ восстановлена L-valueness для совместимых типов что было сломано в регрессии между версиями 2.2.2 и 2.2.4:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                                <font color="#461b7e"># nim cpp
   var x: CppCompatibleType
   takeRef(x)  # Снова работает как lvalue
</font>                              </pre></li>                              <li class="l">Исправления в бэкенде для компиляции в C (refc):
<ul>                                <li class="l">pthread на некоторых платформах: исправлена генерация кода для pthread_mutex_t с использованием .abi;
</li>                                <li class="l">Обобщённые типы с GC-памятью: устранена генерация некорректного C-кода для generic-типов, содержащих управляемую память;
</li>                                <li class="l">Виртуальная машина
<ul>                                  <li class="l">Глобальные переменные и присваивания: множественные исправления работы с глобальными переменными на этапе компиляции. 
</li>                                  <li class="l">Case-объекты из compileTime proc: исправлена передача вариантных объектов как static-параметров.
</li>                                  <li class="l">repr для длинных строк под refc: устранён RangeDefect при использовании repr.
</li>                                  <li class="l">Исправления в стандартной библиотеке
<ul>                                    <li class="l">В strutils.formatSize исправлена работа с большими значениями, близкими к int64.high:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                                      <font color="#461b7e">echo formatSize(9223372036854775807)  # Теперь корректный результат
</font>                                    </pre></li>                                    <li class="l">В deques восстановлена совместимость поведения итератора items между версиями 2.0.16 и 2.2.0.
</li>                                    <li class="l">В lists.SinglyLinkedList.remove устранён AssertionDefect при удалении элементов из односвязного списка.

</li>                                    <li class="l">В tables.withValue исправлено условие проверки в макросе withValue для неизменяемых таблиц.
</li>                                    <li class="l">Прагмы и области видимости
<ul>                                      <li class="l">В &quot;{.push raises: [].}&quot; исправлено некорректное игнорирование лексических областей видимости для push-прагм с raises.

</li>                                      <li class="l">Устранён эффект «утечки»отключения предупреждений за пределы pragma-блоков:
<pre style="overflow-wrap: break-word; word-wrap: break-word; word-break: break-all; word-break: break-word;">                                        <font color="#461b7e">{.push warning[UnusedImport]: off.}
   import module1
   {.pop.}
   import module2  # Предупреждения теперь корректно включены
</font>                                      </pre></li>                                      <li class="l">Прочие важные исправления
<ul>                                        <li class="l">Сравнение cstring: добавлены отсутствовавшие операторы &quot;&lt;&quot; и &quot;cmp&quot; для cstring.
</li>                                        <li class="l">Проверка диапазонов float: включена корректная проверка диапазонов для чисел с плавающей точкой

<td class="ctxt" colspan="2">Для человека структурирование через отступы —естественный приём. Так пишут списки, планы, вложенные элементы.
<p>Конечно, можно так:
<br />if (x &gt;0) {<br />    printf(&quot;Positive\n&quot;);
<br />}else {<br />    printf(&quot;Non-positive\n&quot;);
<br />}</p>                                                      <p>Но я человек, и мне удобнее так:
<br />if x &gt;0:
<br />  echo &quot;Positive&quot;<br />else:
<br />  echo &quot;Non-positive&quot;</p>                                                      <p>Языки с отступами формируют повышенную дисциплина кодирования. Программист не может &quot;сломать стиль&quot;случайными пробелами и код различных авторов в проекте выглядит более единообразно. Кроме того, меньше конфликтов при совместной разработке (например, при слиянии веток в Git).

                           <td class="ctxt" colspan="2">Компилятор - изначально, из исходного кода языка высокого уровня в исходный код языка низкого уровня (gcc -S main.c -o main.asm).
<p>Транспайлер - из исходного кода одного языка в исходный код другого языка (того же уровня). Частный случай компилятора. (Nim -&gt;C)
</p>                                                                  <p>Ассемблер - из исходного кода низкого уровня в машинный код. (Современный компилятор включает ассемблер.)
</p>                                                                  <p>Интерпретатор - разбирает и выполняет программу построчно, минуя компиляцию в машинный код. (Bash, Python, QBasic).
</p>                                                                  <p>Транслятор - общее понятие, включающее и компилятор (в т.ч. транспайлер), и интерпретатор, и ассемблер. Грубо говоря, переводчик кода.
</p>                                                                  <p>( Транслятор &gt;Компилятор &gt;Транспайлер )
